## 질문 로그

### Q1. 현재 GitHub Actions CD 과정에서 원격 서버(OCI 호스트)로 빌드 결과물(app.jar)과 설정 파일을 SCP로 전송하고 SSH로 원격 빌드/배포를 실행하는 아키텍처를 선택했습니다. 이 방식에서 원격 호스트 장애나 빌드 에러 등으로 배포가 실패할 때의 롤백 대안을 어떻게 고려하고 계신가요?

- 막힌 결정: 배포 실패 또는 빌드 실패 시 이전 버전의 컨테이너를 가용 상태로 유지하면서 무중단 또는 저지연으로 되돌리는 롤백 메커니즘이 확정되지 않았습니다.
- 제공된 옵션:
  - 1. (추천) 호스트 측에 이전 배포된 Docker 이미지(`sugang-helper-app:previous`) 또는 빌드 버전 태그(`sugang-helper-app:${GITHUB_SHA}`)를 보존하여 에러 발생 시 즉시 직전 컴포즈 파일과 이전 태그로 롤백 실행
  - 2. 롤백을 자동화하지 않고, GitHub Actions 워크플로우에 이전 성공한 커밋으로 재배포(Re-run)하는 수동 롤백 가이드 문서 마련
  - 3. 완전한 Blue-Green 배포 적용: 두 개의 컨테이너 포트(예: 8080, 8081)를 사용하여 신규 버전 헬스 체크 확인 후 Nginx Proxy Manager의 프록시 패스를 전환하고 기존 포트 컨테이너를 내리는 구조 구현
  - 4. 다른 옵션 더 추천받기
  - 5. 직접 답변
- 답변: 3 -> 블루그린 취소 및 단순 재시작(Recreate) 방식으로 선회
- 결과: 거부
- 해석 결과: 복잡한 블루그린 아키텍처 대신 기존의 심플한 Recreate(컨테이너 내리고 새로 올리기) 방식을 고수하기로 변경함
- 새로 확정된 결정: 무중단 블루그린 배포는 사용하지 않고, 단일 컨테이너 Recreate 방식을 사용한다.
- 남은 열린 질문: Recreate 방식에서 발생하는 짧은 다운타임(Downtime) 허용 및 배포 에러 시의 최소한의 복구 방식

### Q2. 블루그린을 적용하지 않고 기존 컨테이너를 내린 후 새 컨테이너를 띄우는 Recreate 방식을 사용할 때, 배포 실패 시 복구(수동/자동)를 위한 최소한의 안전장치로 무엇을 적용하시겠습니까?

- 막힌 결정: 새 버전 구동 실패 시 자동으로 이전 이미지로 되돌릴지, 아니면 수동 재배포(GitHub Actions Re-run)에만 의존할지 정해지지 않았습니다.
- 제공된 옵션:
  - 1. (추천) 수동 재배포(Re-run) 및 정적 롤백 가이드: 빌드 에러나 런타임 장애 시 GitHub Actions에서 이전 성공 커밋을 재실행하여 되돌리는 방식을 사용하고 추가 복구 자동화 스크립트는 두지 않음
  - 2. 호스트 측 로컬 백업 태그 기반 쉘 스크립트 롤백: `docker build` 전 기존 컨테이너의 이미지를 `sugang-helper-app:backup`으로 태깅해 두고, 새 컨테이너 `up` 후 헬스 체크 실패 시 스크립트가 자동으로 백업 이미지로 복원하고 종료
  - 3. 직접 답변
- 답변: 1
- 결과: 확정
- 해석 결과: 스크립트 복잡성과 자동 롤백 오동작 리스크를 최소화하기 위해, 배포 장애 시 GitHub Actions에서 수동 재배포(Re-run)로 이전 정상 빌드를 복원하는 단순하고 확실한 방식을 선택함
- 새로 확정된 결정: 배포 실패 시 수동 재배포(Re-run) 및 복구 문서 방침을 따른다.
- 남은 열린 질문: 없음

## 확정된 결정

- **배포 모델**: 단일 서버 컨테이너의 기존 인스턴스를 종료하고 새 빌드 본을 실행하는 단순 **Recreate(재시작) 방식**을 고수한다.
- **롤백 정책**: 장애 발생 시 호스트에 복잡한 자동화 스크립트를 두지 않고, GitHub Actions 워크플로우 대시보드에서 직전 성공 커밋에 대해 **수동 재배포(Re-run)**를 트리거하여 복원한다.
- **아키텍처 간소화**: Nginx Proxy Manager의 프록시 설정이나 호스트의 별도 upstream 포트 변경 로직은 일체 반영하지 않음으로써 인프라의 복잡도를 낮게 보존한다.

## 임시 가정

- 배포 중 발생하는 약 수 초 ~ 수십 초 간의 짧은 다운타임은 사용자 이용 패턴 및 오라클 클라우드 프리티어 환경 제약상 허용 가능하다는 가정입니다.

## 최종 정리

### 1. 정리된 방향
- 인프라 복잡성 최소화와 유지보수의 명확함을 목표로 삼아, 서버 배포 방식을 단순 Recreate(기존 중지 후 새 컨테이너 시작)로 통일하고, 장애 시에는 GitHub Actions UI에서 수동 재배포(Re-run)를 진행하도록 설계 방향을 수립했습니다.

### 2. 확정된 결정
- 단일 컨테이너 Recreate 배포 및 GitHub Actions를 통한 수동 재배포 롤백 방침을 확정했습니다.

### 3. 아직 남은 known unknowns
- 없음.

### 4. 핵심 가정
- 배포가 진행되는 동안 발생하는 일시적인 서비스 일시 중단(Downtime)은 수용 가능한 비기능 요구사항 범위에 들어간다는 가정입니다.

### 5. 가장 큰 리스크
- 빌드/구동 장애가 발생하고 운영자가 인지하여 수동 롤백을 실행할 때까지의 장애 지속 시간(Downtime)이 늘어날 리스크가 있습니다.

### 6. 주요 tradeoff
- 무정전 블루그린 배포(Zero-Downtime)를 기회비용으로 포기하는 대신, 오라클 프리티어 환경에서의 극도로 제한적인 메모리 리소스 안정성(스프링 인스턴스 중복 방지)과 배포 쉘스크립트의 극도의 단순함 및 가독성을 얻었습니다.

### 7. 하지 않기로 한 것
- 복잡한 Blue-Green 전환용 Nginx 설정 제어, 호스트 쉘 스크립트 기반의 조건부 자동 롤백 자동화는 진행하지 않기로 결정했습니다.

### 8. 가장 작은 다음 행동
- CI/CD 및 도커 아키텍처에 대한 모든 의사결정이 확정되었으므로, 최종 계획을 검증 로그에 기입하고 세션을 닫습니다.

### 9. 왜 여기서 멈춰도 되는지
- 더 이상 고가치 질문이 없으며, 인프라의 리스크 요인이 모두 수면 위로 드러나 명시적인 Tradeoff 판단이 끝났기 때문입니다.
