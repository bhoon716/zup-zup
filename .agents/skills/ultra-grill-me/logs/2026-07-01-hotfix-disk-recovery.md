# 운영 디스크 포화 임시조치 Grill 세션

- 날짜: 2026-07-01
- 케이스: implementation-plan-grill
- 상태: 완료

## 요약

운영 복구를 최우선으로 두고, `binlog`는 30일 보존하되 MySQL 외부의 Docker JSON 로그와 build cache를 먼저 정리하고, 관측 스택은 임시 중지하는 방향으로 임시조치 범위를 좁혔다.

## 질문 로그

### Q1. 이번 임시조치의 1차 목표를 무엇으로 둘까요?

- 막힌 결정: 서비스 복구와 재발 억제 중 무엇을 우선할지 정해져야 다음 조치의 우선순위를 정할 수 있음
- 답변: 1. 임시 조치는 일단 서비스 복구 우선
- 결과: 확정

### Q2. 운영 복구를 위해 `MySQL binlog`를 즉시 purge하는 쪽으로 가도 될까요?

- 막힌 결정: binlog를 건드려도 되는지 여부가 정해지지 않으면 안전한 복구 범위를 선택할 수 없음
- 답변: 너무 오래된 건 삭제해도 되지만 최근 에러 부분은 살리고 싶음
- 결과: 임시 가정

### Q3. `binlog`를 몇 일치 남길까요?

- 막힌 결정: purge 범위를 수치로 정해야 실제 정리 대상이 확정됨
- 답변: 30일
- 결과: 확정

### Q4. 첫 임시 정리 대상으로 무엇을 둘까요?

- 막힌 결정: `binlog`를 보존한다면 어떤 비-binlog 영역부터 정리할지 정해야 함
- 답변: Docker JSON 로그와 build cache 먼저 정리
- 결과: 확정

### Q5. 이 서버에서 `binlog`를 복제나 포인트인타임 복구에 실제로 사용하나요?

- 막힌 결정: 복제/PITR 사용 여부에 따라 보존 정책이 완전히 달라짐
- 답변: 아니오, 운영 복구 우선이라 거의 다 purge 가능
- 결과: 확정

### Q6. 최근 `binlog`를 최소 며칠치 남길까요?

- 막힌 결정: purge 범위를 안전하게 자를 최소 보존 기간이 필요함
- 답변: 아 그래도 30일
- 결과: 확정

### Q7. 임시 복구 동안 `Promtail/Loki/Grafana` 같은 관측 스택을 잠깐 중지해도 될까요?

- 막힌 결정: 로그가 계속 쌓이는 경로를 멈출지 여부가 정해져야 즉시 복구 효과를 높일 수 있음
- 답변: 가능, 관측 잠깐 중지하고 서비스 복구를 최우선
- 결과: 확정

## 확정된 결정

- 임시조치의 1차 목표는 서비스 복구다.
- `binlog`는 30일 보존 방향으로 간다.
- 즉시 정리 대상은 Docker JSON 로그와 build cache다.
- 관측 스택은 임시로 중지할 수 있다.

## 임시 가정

- `binlog`는 복제나 PITR에 사용하지 않는다.

## 남은 known unknowns

- 어떤 Docker 로그/캐시 정리 명령을 실제로 실행할지
- 관측 스택을 어디까지 중지할지

## 최종 정리

운영 복구용 임시조치는 `binlog`를 건드리지 않고 Docker JSON 로그와 build cache를 먼저 정리한 뒤, 관측 스택을 잠시 내려 디스크 사용을 즉시 낮추는 방향으로 정리됐다.
