# 아키텍처 지식 베이스 (Architecture Knowledge Base)

본 문서에는 수강신청 헬퍼 프로젝트를 개발 및 개선할 때 지속적으로 준수하고 활용해야 하는 영속적 지침 및 의사결정 규칙이 기록됩니다.

## 1. 수강평(리뷰) 그룹화 정책
* **규칙**: 강의 리뷰(`CourseReview`)는 매 학기마다 갱신 생성되는 개별 강의 고유 키(`courseKey`)에 일대일 매핑되지 않으며, `(subjectCode, professor)` 복합 키를 기준으로 논리 그룹화됩니다.
* **이유**: 학생들이 다음 학기 수강 계획을 짤 때 과거에 작성된 동일 과목 및 교수 조합의 유의미한 강의 평가들을 한곳에서 모아보기 위함입니다.
* **코딩 가이드**: 새로운 강의 리뷰를 적재 또는 수정할 때 `ReviewScopeKey` 구조체를 사용해 해당 조합을 검출하고 통계를 갱신해야 합니다.

## 2. 보안 통신 무상태성(Stateless) 강제 규칙
* **규칙**: 백엔드 스프링은 무상태(Stateless) 상태를 지향하며, 토큰은 HttpSession 영역에 영구 유지되어 브라우저에 세션 ID로 전파되어서는 안 됩니다.
* **이유**: 분산 환경에서의 확장을 수월하게 하고 세션 하이재킹에 따른 인증 정보 노출 가능성을 차단하기 위함입니다.

## 3. 데이터 논리 삭제(Soft Delete) 표준 정책
* **규칙**: `BaseEntity`를 상속하여 `deletedAt` 컬럼으로 삭제 상태를 기록하는 모든 도메인 데이터 클래스는 기본 쿼리에서 자동 차단되어야 합니다.
* **코딩 가이드**: 신규 엔티티 추가 시 반드시 `@SQLRestriction("deleted_at IS NULL")` 어노테이션이 동반 매핑되어 작동하는지 검사하십시오.
