## 질문 로그

### Q1. 새로고침 시 로그인하지 않은 일반 사용자의 네비게이션바 화면에 로그인 전용 메뉴(내 시간표, 알림/구독 등)를 어떻게 처리하는 것이 바람직할까요?

- 막힌 결정: 네비게이션바 로딩 개선 설계에 있어서, 로그인/비로그인 사용자의 초기 렌더링 기대 동작에 대한 구체적인 기능 요구사항이 정의되지 않았습니다.
- 제공된 옵션:
  - 1. 쿠키 기반 힌트 로그인 판단: 브라우저에 쿠키(예: `is_logged_in`)를 확인하여, 쿠키가 없으면 즉시 비로그인 메뉴만 보이고, 쿠키가 있으면 세션 유효성 검증 중 스켈레톤을 렌더링
  - 2. 단순 지연 스켈레톤 렌더링: 로딩을 150~200ms 정도 지연시킨 후 API가 완료되지 않은 경우에만 스켈레톤을 렌더링하여 빠른 네트워크에서는 스켈레톤 노출 방지
  - 3. 로컬스토리지 기반 힌트 판단: LocalStorage의 로그인 플래그를 읽되, 클라이언트 하이드레이션 완료 후에만 판단하도록 적용
  - 4. (추천) 가드가 적용된 메뉴 상시 노출 (스켈레톤 불필요): 로그인 유무에 상관없이 메뉴들을 상시 노출하되, 비로그인 상태에서 클릭 시 로그인 유도 모달을 표시. 로딩 여부와 무관하여 깜빡임과 레이아웃 시프트가 완벽히 제거됨.
  - 5. 기본 비인증 레이아웃 렌더링 (CSS 은닉): 로딩 중에는 비인증 메뉴 구성만 노출하며 스켈레톤을 그리지 않고, 세션 확인 완료 시점에 로그인 상태이면 로그인 메뉴를 페이드인 등의 연출로 띄움.
  - 6. 다른 옵션 더 추천받기
  - 7. 직접 답변
- 답변: 1
- 결과: 확정
- 해석 결과: SSR 환경에서의 레이아웃 깜빡임 및 레이아웃 시프트를 완벽하게 방지하기 위해, 클라이언트와 서버에서 동시에 감지할 수 있는 가벼운 쿠키 기반의 로그인 힌트 플래그를 도입하기로 확정함.
- 새로 확정된 결정:
  - 로그인 상태 힌트를 위한 독립적인 비독점 쿠키 플래그(예: `is_logged_in`)를 도입하여, 해당 플래그가 없으면 세션 확인 API 로딩 대기 중에도 스켈레톤을 렌더링하지 않고 즉시 비로그인 기본 레이아웃을 표시한다.
- 남은 열린 질문:
  - 로그인 힌트 쿠키의 접근 권한 제약 및 세션 불일치 예방 (Q2)
  - 테스트 및 유효성 검증 방식

### Q2. 로그인 힌트 플래그용 쿠키(`is_logged_in`)의 보안 속성(JS 접근 가능 여부 등)과 실제 서버 세션과의 만료 주기 불일치(동기화) 문제를 어떻게 해결할까요?

- 막힌 결정: 클라이언트 렌더링 및 SSR 동시 인지를 위한 쿠키의 노출 범위 수준과 세션 불일치 시의 예외 처리(장애 모드) 방식이 확정되지 않았습니다.
- 제공된 옵션:
  - 1. (추천) 클라이언트 JS 조회 허용 쿠키(`httpOnly=false`) 및 실패 피드백 동기화: 클라이언트 컴포넌트에서도 `document.cookie`로 즉시 상태를 확인하도록 `httpOnly`를 해제하고, 백엔드 세션 검증(`checkSession`) 실패 시나 로그아웃 시점에 API 및 클라이언트 코드 양쪽에서 쿠키를 명시적으로 삭제함.
  - 2. 미들웨어/서버 컴포넌트 전용 판별 (`httpOnly=true` 유지): 보안을 위해 `httpOnly`를 유지하고, Next.js Middleware 또는 루트 서버 컴포넌트 단에서만 쿠키를 읽어 Header 컴포넌트의 초기 props로 전달. 클라이언트 JS에서의 직접 조작은 원천 차단함.
  - 3. 다른 옵션 더 추천받기
  - 4. 직접 답변
- 답변: 1
- 결과: 확정
- 해석 결과: XSS 리스크가 없으면서 기민하게 동기화가 가능하고 Next.js SSR 및 정적 최적화를 방해하지 않는 `httpOnly=false` 쿠키 방식을 채택함.
- 새로 확정된 결정:
  - 힌트용 `is_logged_in` 쿠키는 JS에서 읽기 가능하도록 `httpOnly=false`로 발급한다.
  - 로그인 성공 시 서버 측 응답(Set-Cookie) 혹은 클라이언트 로그인 완료 콜백을 통해 쿠키를 심는다.
  - 로그아웃 및 백엔드 인증 에러 발생 시, 클라이언트 핸들러와 API 서버 양쪽에서 쿠키를 명시적으로 파괴하여 세션 상태 불일치를 즉시 해결한다.
- 남은 열린 질문:
  - 힌트 기반 네비게이션바 렌더링 및 동기화 테스트 검증 전략 (Q3)

### Q3. 새롭게 추가되는 힌트 쿠키 기반의 네비게이션바 렌더링 분기 로직과 쿠키 동기화 로직을 어떻게 검증(테스트)할까요?

- 막힌 결정: 새로 도입될 쿠키 플래그 분기 동작의 기능적 회귀 방지를 위한 테스트 설계 수준이 확정되지 않았습니다.
- 제공된 옵션:
  - 1. (추천) 컴포넌트 단위 테스트(Vitest) 및 모의 쿠키 상태 검증 자동화: 기존 `nav-links.test.tsx` 테스트 코드에 `document.cookie`를 모킹(Mocking)하는 테스트 케이스를 추가하여 쿠키 존재 여부 및 `isLoading` 조합에 따라 스켈레톤과 메뉴 렌더링이 정상 분기되는지 자동화 검증함.
  - 2. 로컬 브라우저 수동 통합 테스트 검증: 자동화 테스트를 추가하지 않고 로컬 개발 서버를 띄워 수동으로 로그인/로그아웃/새로고침을 반복하며 크롬 개발자 도구로 수동 확인 및 UI 시각 검수 진행.
  - 3. 다른 옵션 더 추천받기
  - 4. 직접 답변
- 답변: 1
- 결과: 확정
- 해석 결과: 기존 Vitest 테스트 모듈 내에 document.cookie 모킹 케이스를 추가하여 신규 쿠키 분기 로직에 대한 회귀 방지 자동화 테스트를 확립하기로 함.
- 새로 확정된 결정:
  - `nav-links.test.tsx` 파일 내에서 쿠키 힌트 상태에 따른 조건부 렌더링에 대한 단위 테스트 코드를 작성 및 실행한다.

## 확정된 결정

- **로그인 상태 힌트용 독립 쿠키 도입**: 브라우저 쿠키(예: `is_logged_in`)를 확인하여 최초 진입 시 스켈레톤 노출 여부를 동적으로 제어 (쿠키가 없는 비로그인 상태일 때는 세션 체크 중에도 스켈레톤을 생략하고 즉시 비로그인 기본 레이아웃 노출).
- **보안 속성 및 JS 노출 허용**: 단순 상태 힌트 용도이므로 XSS 세션 탈취 리스크가 없다고 보고 `httpOnly=false` 및 `secure=true` (운영 환경)로 발급하여 클라이언트 JS 접근을 허용함.
- **양방향 동기화 및 즉시 파괴**: 백엔드 세션 만료 응답(401 Unauthorized), 세션 검증 실패(`checkSession` 에러), 또는 로그아웃 시점에 API 서버 및 클라이언트 측 모두에서 이 쿠키를 즉시 만료/삭제하여 불일치에 따른 장애 모드를 예방함.
- **Vitest 단위 테스트 자동화**: `nav-links.test.tsx`에서 `document.cookie` 상태를 모킹하여 힌트 쿠키와 `isLoading` 값 조합에 따른 렌더링 동작을 자동화된 테스트로 보호함.

## 임시 가정

- 로그인 힌트 쿠키의 만료 시간(Max-Age)은 백엔드 세션 쿠키(Refresh Token 등)의 최대 수명(예: 30일 등)과 매칭되도록 넉넉하게 설정하여, 일반적인 로그인 유지 기간 동안 힌트가 유지되도록 가정함.

## 남은 known unknowns

- 기존 백엔드에서 쿠키를 발행할 때 이 힌트 쿠키를 동시 발급하도록 API 응답 로직(Spring Boot 서버)을 연동하는 부분의 수정 범위.

## 최종 정리

### 1. 정리된 방향
새로고침 시 비로그인 사용자에게 무분별한 스켈레톤 렌더링(레이아웃 깨짐)이 노출되는 문제를 해결하기 위해, 보안상 무해하고 클라이언트 JS 조회가 용이한 비인증 상태 힌트용 독립 쿠키(`is_logged_in`)를 도입하여 초기 렌더링을 최적화합니다.

### 2. 확정된 결정
- `httpOnly=false` 속성의 `is_logged_in` 쿠키 도입.
- 로그인 성공 시 서버 및 클라이언트에서 쿠키를 생성하고, 로그아웃/만료 시 양방향에서 즉시 삭제하여 동기화 보장.
- Vitest 자동화 단위 테스트를 추가해 `document.cookie` 모킹으로 렌더링 안정성 보호.

### 3. 아직 남은 known unknowns
- 백엔드 Spring Boot(Java)의 로그인/토큰 갱신 응답 시 이 `is_logged_in` 쿠키를 동시 발행하는 컨트롤러/쿠키 유틸리티 코드의 수정 범위 파악 필요.

### 4. 핵심 가정
- `is_logged_in` 쿠키는 단순한 시각적 힌트 용도로만 기능하고 실제 인가(Authorization)에는 영향을 주지 않으므로 `httpOnly=false`여도 안전함.

### 5. 가장 큰 리스크
- 로그인 힌트 쿠키 삭제 처리가 에러 핸들러나 미들웨어 등에서 누락될 경우, 비로그인 상태인데 로그인 상태인 것처럼 메뉴를 렌더링하려 하다가 세션 만료로 튕기는 등의 일시적 상태 불일치 리스크가 있음.

### 6. 주요 tradeoff
- Next.js Middleware 복잡도를 피하기 위해 `httpOnly=false`로 클라이언트의 조회를 허용하되, XSS 취약점 우려를 최소화하기 위해 해당 쿠키에는 단순 플래그 정보만 담도록 하여 복잡성과 보안 사이의 타협점을 찾음.

### 7. 하지 않기로 한 것
- Next.js Middleware를 통한 `httpOnly=true` 쿠키 주입 및 서버 사이드 Route 차단 (구현 복잡도 증가 방지).
- LocalStorage를 통한 힌트 구현 (SSR 단계에서의 하이드레이션 불일치와 깜빡임 예방을 위함).

### 8. 가장 작은 다음 행동
- 백엔드(Spring Boot)에서 로그인 성공 및 삭제 시 `is_logged_in` 쿠키를 다루는 유틸리티 메서드 추가/수정 범위 및 프론트엔드(Next.js) `Header`와 `NavLinks`에서 이 쿠키를 꺼내 렌더링 조건에 결합하는 코드를 분석하고 구현 계획 수립.

### 9. 왜 여기서 멈춰도 되는지
- 시각적 깜빡임을 유발하는 타이밍 버그의 완화 설계 방향, 보안 tradeoff, 장애 예방 및 테스트 전략까지의 설계 의사결정이 모두 명확하게 종결되어 기술 설계상 더 이상 중요한 질문이 없으므로 검증 완료를 선언함.
