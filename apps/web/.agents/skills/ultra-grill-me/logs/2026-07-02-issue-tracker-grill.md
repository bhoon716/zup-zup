# 이슈 트래커 설계 Grill 세션

- 날짜: 2026-07-02
- 케이스: architecture-decision-grill
- 상태: 진행 중

## 요약

레포 내부 파일 기반 이슈 트래커를 만들되, 상태 모델과 아카이브 경계를 먼저 확정하는 중입니다.

## 질문 로그

### Q1. 이슈를 실제 GitHub Issues와 연동할까요, 아니면 레포 내부 파일 기반 이슈 트래커로 둘까요?

- 막힌 결정: source of truth가 GitHub인지 레포 파일인지 정해지지 않음
- 답변: 레포 내부 파일 기반 이슈 트래커
- 결과: 확정

### Q2. 종료 이슈를 `issues/` 안에 그대로 두고 상태만 바꿀까요, 아니면 `issues/closed/` 같은 분리 디렉터리를 둘까요?

- 막힌 결정: 종료 이슈의 물리적 보관 위치가 정해지지 않음
- 답변: `issues/closed/` 분리, 더 다양한 상태도 허용
- 결과: 확정

### Q3. `review`를 별도 상태로 둘까요?

- 막힌 결정: 상태 집합을 더 단순하게 압축할지, 검증 단계를 명시적으로 둘지 정해지지 않음
- 답변: 대기 중
- 결과: 진행 중

## 확정된 결정

- 이슈 관리의 source of truth는 레포 내부 파일이다.
- 상태 모델은 `open / in-progress / blocked / closed`로 간단하게 유지한다.
- 종료 이슈는 `issues/closed/`에 둔다.
- `review`는 별도 1급 상태로 두지 않는다.

## 임시 가정

- `archive/`는 과도기적 백업으로만 유지할 수 있다.

## 남은 known unknowns

- 새 이슈를 자동 생성하는 helper를 둘지 여부
- 기존 issue log의 더 오래된 legacy 참조를 추가 정리할지 여부

## 최종 정리

레포 내부 파일 기반 이슈 트래커를 채택했고, 상태는 `open / in-progress / blocked / closed`로 유지한다. 아카이브된 이슈와 로그는 `issues/closed/`와 `logs/`로 복원했다.

## 확정된 결정

- 이슈 관리는 레포 내부 파일 기반으로 한다.
- 종료 이슈는 별도 디렉터리로 분리한다.

## 임시 가정

- `archive/`는 과도기적 보관소이며 장기적으로 제거될 가능성이 높다.

## 남은 known unknowns

- 어떤 상태 집합을 1급으로 둘지
- 이슈 파일의 표준 스키마를 어떻게 둘지
- 복원한 flowness 이슈/로그를 새 구조에 어떻게 매핑할지
